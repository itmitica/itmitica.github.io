Syntax highlighting

Debugging can't get simpler than that.

User and developer reliability

A few moths back I've put together a small SQL script for a colleague of mine to run at his discretion. Things went smooth for a while, my colleague, although not a professional, he is sharp and a fast learner.

Yesterday he complained to me that the script stopped returning results for a few days now, which was odd, and this was affecting his reporting.

As I understood later, part of the problem was that he was only acknowledging the final "success" messages and he was ignoring essential intermediary messages for the script.

In hindsight, I probably should have parameterized the query instead of forcing him to change the script source directly. In my defense, I'm required to come up with custom queries on a daily basis and often on the spot and under the pressure of time.

The tools

Anyway, all he had to do was to change some date values in it and then run the script. He used notepad for the task.

There are some voices against it, and Oracle tools like SQL*Plus or Forms do not have syntax highlighting capabilities, but, given the choice, I'm always looking for code editors that do. As such, I've asked him to send me the script and as soon as I opened it I already found the problem, I had no need to run it to debug it.

User feedback reliability

One final point I'd like to make is about user feedback reliability. He didn't acknowledge the error displayed during the script running, for him it was only the final messages that mattered. Something like this:

blah blah blah
blah.....
blah ERROR:
blah ORA-01756: quoted string not properly terminated
blah.....
blah blah blah
FINALLY!!! n rows updated.
FINALLY!!! Commit complete.

Luckily for me, I've learned about it many years ago.
